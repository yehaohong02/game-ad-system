6.1.2 数据清洗与质量监控
重复/异常点击检测：利用 ClickHouse 的 clicks_spam 列，标记同一 IP 短时间高频点击，聚合时剔除。

缺失值处理：revenue 缺失时用 AppsFlyer 的 predicted_ltv 或同类 campaign 均值填充；installs 为0但花费>0 的条目单独标记。

数据延迟补偿：AppsFlyer 收入数据滞后1-3天，采用滑动窗口归因修正，将收入分配至安装日。

每日质量检查SQL：

sql
SELECT date,
       countIf(installs = 0 AND spend > 0) AS anomaly_spend_no_installs,
       countIf(revenue < 0) AS negative_revenue_count
FROM ads_performance
WHERE date = today()
若异常占比 > 5%，触发人工审查。

6.1.3 异常检测算法
规则：

花费异动：近3天日均花费 > 历史7日均值 + 2σ，且 ROAS < 0.6 → critical

CPI 飙升：单日 CPI > CPI_cap × 1.5 且连续2天

ROAS 暴跌：7日滑动平均 ROAS < target_roas × 0.5，且花费 > $100

数据缺失：>2天无数据

实现 (src/data/anomaly.py)：

python
def compute_zscore(series):
    mean = statistics.mean(series[-7:])
    stdev = statistics.stdev(series[-7:]) if len(series)>=7 else 0
    return (series[-1] - mean) / (stdev + 1e-6)
结合阈值写入 alerts 表。

6.1.4 实时事件处理
FastAPI 端点接收 Meta/AppsFlyer webhook，发布到 Redis Streams。

消费者监听流，更新 ClickHouse 物化视图，并检查分钟级花费是否在10分钟内超全天预算40%，触发熔断。

6.2 创意模块 (Creative)
6.2.1 精细化标签体系
视觉标签：使用 CLIP (ViT-L/14@336px) 对采样帧推理，输出带置信度的标签（如 {"label": "真人剧情", "confidence": 0.87}），过滤概率 >0.3 的 top5。

语音/文案标签：Whisper 转录音频，正则 + 实体识别提取关键短语（如“免费100抽”），分析语音情感、语速，文案转为 embedding 用于相似度匹配。

存储：ClickHouse creative_tags 表，visual_tags 等列使用 Array(String)，建 Bloom filter 索引。

6.2.2 标签与效果归因
关联 SQL（每日）：

sql
SELECT t.visual_tags, t.audio_tags, t.text_keywords,
       avg(p.ctr) as avg_ctr, avg(p.cpi) as avg_cpi, avg(p.roi) as avg_roas
FROM ads_performance p
JOIN creative_tags t ON p.creative_id = t.creative_id
WHERE p.date >= today() - 7
GROUP BY t.visual_tags, t.audio_tags, t.text_keywords
边际贡献计算：对创意特征使用 GBM 模型预测 ROAS，计算 Shapley Value，量化每个标签的独立贡献及交互效应。

6.2.3 AI 素材生成集成
创意简报：LLM 基于高潜标签生成结构化简报（视觉建议、文案方向、时长）。

视频混编：接入 Runway/自建 AnimateDiff 管线，输入参考素材 + 标签组合 + LLM 生成的脚本，自动输出新 creative_id 并纳入测试池。

6.3.3 Bid Agent 调整策略
基于目标 CPI 的 PID 控制器：Δbid = Kp*e + Ki*∫e dt + Kd*de/dt，误差 e = current_cpi - target_cpi，限制单次调整 ±20%。

新广告冷启动出价：基于类似标签素材的历史平均 CPI，上浮15%获取初始量。

6.3.4 沙盒模拟
提取过去30天小时级数据，训练 LightGBM 时序模型预测不同操作（暂停、调价）后48小时的 ROAS 和花费。

Agent 调用真实 API 前先查询模拟结果，只有预期改善概率 >70% 才执行。

6.3.5 AB 测试框架
新 campaign 按国家、平台分层采样，随机分配至控制组（规则引擎优化）和实验组（Agent 优化）。

收集7日数据后计算 ROAS 差异的 p-value，若 p<0.05 自动推广优胜策略。

6.4 安全模块 (Safety)
6.4.1 L1 预算硬限制
Redis 哈希存储每日消耗：budget:daily:2025-06-01，字段为 account_123 等。

Lua 脚本原子检查：

lua
local current = redis.call('HGET', KEYS[1], ARGV[1])
current = current and tonumber(current) or 0
local new_val = current + tonumber(ARGV[2])
local limit = tonumber(ARGV[3])
if new_val <= limit then
    redis.call('HSET', KEYS[1], ARGV[1], new_val)
    return 1
else
    return 0
end
返回0则拒绝操作。

6.4.2 L2 操作熔断器
Redis ZSET 滑动窗口计数，键 cb:{agent_id}:{action}。

10分钟内同操作失败 ≥3 次则熔断5分钟（SET cb:blocked:{key} 1 EX 300）。

6.4.3 L3 参数校验
出价变化率 ≤ 50%，单广告日预算 ≥ $5，广告组预算总和 ≤ campaign 总预算。

货币单位检查：基于账户配置，判断数值是否在合理范围。

6.4.4 语义校验
使用轻量 LLM 二次确认操作合理性，如判断出价 
0.05
与历史出价区间
0.05与历史出价区间1.5-$2.8 是否匹配，不通过则挂起人工审核。

6.4.5 分布式锁
修改同一广告对象时使用 Redis 锁 lock:ad:{ad_id}，确保串行化。

6.5 记忆模块 (Memory)
6.5.1 案例向量化
campaign 结束或满7天，由 summarizer.py 生成案例 JSON。

用 text-embedding-3-large 对案例文本（目标+国家+标签+决策+教训）生成3072维向量，存入 ChromaDB，附过滤元数据（country、platform、roas、tags）。

6.5.2 检索增强生成 (RAG)
查询文本示例："Objective: ROAS>=0.8, Country: US, Tags: 真人剧情, 宝箱"。

返回 top 3 案例的 lessons_learned 和 key_decisions，注入 Agent 系统提示词。

6.5.3 因果效应提取
利用 CausalImpact 库进行干预分析（如暂停低ROAS广告对整体ROAS的影响）。

代码示例：

python
from causalimpact import CausalImpact
ci = CausalImpact(data, pre_period, post_period)
ci.run()
effect = ci.summary()
结果存入案例的 causal_effects 字段，如 {"pause_low_roas_ad": 0.18}。

6.5.4 动态案例更新
长期运行的 campaign 每7天生成一个“微案例”，保持记忆新鲜度。
